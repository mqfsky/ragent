## 10 个高频追问与回答

### 1. RAG 为什么不只是“检索 + 生成”？

因为真正能用的 RAG 至少还包含数据清洗、文档解析、分块、查询重写、意图识别、召回融合、排序、会话记忆、可观测性和模型容错。检索和生成只是中间两步。

### 2. 为什么向量检索不够？

向量检索适合语义相似，但在领域限制、精确过滤和排序质量上并不总够用。企业场景需要把意图定向、兜底召回和 Rerank 一起考虑。

### 3. MCP 在这个项目里扮演什么角色？

MCP 让系统不只会查知识，还能按意图调用工具。也就是说，用户问题不一定都走知识检索，有些会转成工具调用，这更接近 Agent 能力。

### 4. 会话记忆为什么重要？

多轮对话里用户经常省略上下文，如果不做历史保留和摘要压缩，模型要么不知道你在问什么，要么上下文过长导致成本过高。

### 5. 模型不稳定怎么处理？

项目通过候选模型、优先级、失败降级和断路器状态控制来处理。某个模型连续失败后会熔断一段时间，再试探恢复。

### 6. 为什么说 Trace 很重要？

RAG 链路比普通 CRUD 长很多，检索、排序、流式输出都可能出问题。没有 Trace 就难以定位瓶颈和异常点。

### 7. 为什么用 TTL？

因为普通 `ThreadLocal` 在异步线程池中容易丢上下文，而 RAG 系统里异步执行很多。TTL 可以把 `traceId`、用户信息等透传过去。

### 8. 这个项目最大的工程价值是什么？

它把知识侧、问答侧、模型侧和运维侧串起来了，体现的是完整 AI 系统的工程落地能力，而不是单个算法点。

因为我没有把自己包装成长期维护者，而是基于真实代码做了系统拆解、调用链跟踪和设计验证。我会强调跨语言迁移能力和对工程系统的学习速度。

## Java 高频问题速记卡

### Spring Boot 启动流程

- `main` 方法里调用 `SpringApplication.run`
- 自动配置装配 Spring 容器
- 扫描组件、Mapper、配置类
- 启动 Web 容器并暴露接口

### Controller / Service / Mapper 分层

- `Controller`
  接收请求、参数校验、返回结果
- `Service`
  承接业务逻辑、流程编排
- `Mapper`
  和数据库做持久化交互

### IOC / AOP

- `IOC`
  让对象由容器管理，减少手动 new 和耦合
- `AOP`
  适合做日志、鉴权、限流、Trace 这类横切逻辑

### 线程池与异步

- 线程池解决任务隔离、资源复用和并发控制
- AI 项目里常见于检索并行、流式输出、异步消息处理
- 上下文透传是异步链路里的常见坑

### Redis / MQ / 数据库在项目里的角色

- `PostgreSQL`
  核心业务数据
- `Redis`
  状态、限流、缓存、协调
- `RocketMQ`
  耗时任务异步化
